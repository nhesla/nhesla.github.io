// Dev Notes
What went well

The ground shader uses the same noise and color parameters as the grass, so the two blend together naturally without any manual color matching.

Several rendering techniques came together well on this one — see the How It Works section below for a full breakdown.

What I learned

Working with GLSL was new territory for me — it's less of a traditional programming language and more of a set of instructions you write directly for the GPU, with its own rules around types, precision, and what data is available at each stage of the pipeline. Being able to describe the techniques I wanted to implement to Claude and have it translate them into working shader code was a huge help. I could focus on understanding what the code was doing and why rather than getting stuck on syntax.

The core idea came from a game programming video about drawing thousands of grass blades in a single draw call. I thought it was a brilliant trick and wanted to figure it out myself.

I also learned that WebGL behavior varies more across GPU vendors than I expected. Something that looked fine on my machine rendered as flickering static on Windows hardware due to differences in how drivers handle back-face rendering. That's the kind of thing you can't catch without testing on different devices.

Mistakes & what I'd do differently

Grass geometry is rebuilt on the CPU in JavaScript whenever parameters change, which causes a noticeable hitch on sliders like blade count. The right approach would be GPU-side generation using transform feedback — the CPU would just send parameters and the GPU would generate and buffer its own geometry. I'd also add shadow casting between blades and ambient occlusion at the base, which would make the grass feel grounded rather than floating on the surface.

How It Works

01 Single Draw Call via Triangle Strip
Rendering

All grass blades — thousands of them — are submitted to the GPU in a single drawArrays(TRIANGLE_STRIP) call. Individual blades are separate strips, but they're stitched end-to-end using degenerate triangles: zero-area triangles formed by repeating a vertex that the GPU clips away as invisible, creating a seamless invisible bridge between blades.

vs. one draw call per blade:
draw calls → O(1) instead of O(n)
02 Hilbert Curve Memory Layout
CPU Performance

Grass blades are generated by walking a Hilbert curve over the patch grid rather than row by row. A Hilbert curve is a space-filling curve that visits every cell while keeping spatially nearby cells close together in the traversal order. This means blades that are near each other in the world end up near each other in the vertex buffer, which improves cache hit rates when the GPU reads geometry during rendering.

locality: adjacent in world ≈ adjacent in memory
03 Chunked Streaming
Scene Management

The world is divided into fixed-size chunks arranged in a ring around the camera. As the camera moves, chunks that scroll out of range are recycled and rebuilt at the new leading edge rather than allocated fresh. Dirty chunks are rebuilt one per frame to spread the CPU cost out over time rather than spiking all at once when the view changes.

rebuild cost: amortized 1 chunk/frame
memory: fixed pool, no growth
04 Level of Detail (LOD)
GPU Performance

Each chunk stores three versions of its geometry at decreasing segment counts (5, 3, and 1 segments per blade). At render time each chunk samples its distance from the camera and binds the appropriate buffer. Distant grass that would look identical at high detail gets drawn with far fewer triangles, freeing up GPU bandwidth for the foreground.

LOD 0 (near): 5 segments/blade
LOD 1 (mid): 3 segments/blade
LOD 2 (far): 1 segment/blade
05 Frustum Culling
GPU Performance

Before submitting any chunk for drawing, its bounding box is tested against the six planes of the camera's view frustum. Chunks outside the frustum are skipped entirely — no buffer bind, no draw call. The frustum planes are extracted from the combined view-projection matrix each frame, so they automatically track camera movement and rotation.

tested: 6 planes × AABB per chunk
result: ~50% cull rate at typical FOV
06 Vertex Shader Wind
GLSL

Wind is computed entirely on the GPU in the vertex shader — no CPU involvement per frame. Each vertex is displaced by a sine wave that factors in world position, time, and a bladeT value (0 at base, 1 at tip) so the base stays anchored and the tip sways freely. A second turbulence term at a different frequency adds irregularity so blades don't all move in perfect lockstep.

displacement: sin(pos·dir + time·speed) × intensity × bladeT
07 Per-Blade Color Variation
GLSL

Each blade gets its own hue and brightness offset computed in the fragment shader from a hash of its base world position. No texture lookups, no extra uniforms — the hash turns the (x, z) base coordinate into a stable pseudo-random value that's consistent across frames. Hue variance and brightness variance are exposed as uniforms so the controls panel can tune how much color diversity the field has.

hash input: blade base (x, z)
output: hue offset, lightness offset
08 Patchiness via Noise Density
Procedural

Rather than placing blades uniformly, each candidate cell is evaluated against a smooth noise field. The patchiness parameter sharpens the noise curve, biasing placement toward high-density clusters with bare patches between them. At zero patchiness the distribution is uniform; at maximum it produces tight clumps separated by open ground — useful for seaweed and fur presets.

acceptance = densityBias + pow(noise, sharpness) × p
09 Surface-Normal Blade Alignment
Procedural

Each blade is oriented to grow perpendicular to the terrain surface rather than straight up in world space. A surface normal is sampled at the blade's base by finite differencing the heightmap in four directions. That normal is used to build a local coordinate frame (up, right, forward) which rotates the blade's geometry to match the slope, so blades on hills lean naturally with the terrain rather than clipping through it.

normal ← finite diff of heightmap
basis ← Gram-Schmidt orthogonalize
